# KLEE-style Symbolic Execution vs. Random Testing — Coverage Study

## 1. Goal

Re-implement the **core method of KLEE** (Cadar, Dunbar & Engler, USENIX
OSDI 2008, *KLEE: Unassisted and Automatic Generation of High-Coverage
Tests for Complex Systems Programs*) in a small Python / z3 framework and
use it to compare **branch coverage** between:

* (a) **Symbolic execution** (DSE-style path enumeration, with
    `z3-solver` deciding feasibility and producing concrete witnesses).
* (b) **Random testing** (≥ 1 000 uniform random inputs per run).

on a small target function `f(x)` that contains a single
**hard-to-reach branch** satisfied only by the magic value `12345`.

The two central questions:

1. Does the symbolic engine achieve (near) **complete branch coverage**
   and reach the magic branch?
2. Does random testing, in a comparable budget, **miss** the magic
   branch and end up with measurably lower coverage?

The answer to both is "yes" — see §5.

---

## 2. Target function

`target.py` defines a function with **5 `if` statements** = **10
`if`-true / `if`-false edges** that are tracked for coverage:

| Branch | Source-level test                  | Comment                                       |
|--------|------------------------------------|-----------------------------------------------|
| B0     | `x == 12345`                       | **Magic / rare** branch — the hard one        |
| B1     | `x < 0`                            | sign split                                    |
| B2     | `x % 2 == 0`                       | parity split                                  |
| B3     | `x > 1000`                         | magnitude split                               |
| B4     | `x % 3 == 0`                       | divisibility split                            |

The function takes a single integer `x`.  When `x == 12345` it short-
circuits to the "rare" return path.  The branch tracker records every
`(branch_id, taken_bool)` pair the concrete execution passes through; an
edge is considered **covered** if at least one test input has ever
exercised it.

The input domain is `[-1 000 000, 2 000 000]`, i.e. `3 000 001` distinct
integers.  The probability of any single uniform draw hitting the magic
value is therefore `1 / 3 000 001 ≈ 3.33 × 10⁻⁷`.

---

## 3. Symbolic-execution engine (the KLEE analogue)

`symex.py` is a small, transparent re-implementation of the KLEE idea
that the paper describes in §2.2:

> "When program execution branches followed on a symbolic value, the
> system (conceptually) follows both branches, on each path maintaining
> a set of constraints called the *path condition* which must hold for
> execution of that path.  When a path terminates or hits a bug, a test
> case can be generated by solving the current path condition for
> concrete values."

Our engine keeps it minimal:

* One symbolic input variable `x : z3.Int`.
* A depth-first search over the boolean choices at every branch.
* At each branch we ask Z3 whether `path_condition ∧ taken` is
  satisfiable.  If yes we recurse with the augmented condition; if no
  we prune that sub-tree.
* When all branches have been resolved (i.e. the function returned) we
  ask Z3 for a concrete witness and emit it as a test case.

The function-under-test itself lives in `target.py`.  We deliberately
*do not* parse or interpret Python bytecode; we hand-write a parallel
`branch_conditions(x, depth)` table that returns the (taken, not-taken)
Z3 expressions for the same five `if` statements the concrete
implementation contains.  This is the same "symbolic interpretation"
approach KLEE performs on LLVM instructions, just at a much higher level.

Concretely the 5 branches produce a path space of up to `2⁵ = 32`
leaves; constraint solving prunes 19 of them, leaving **13 feasible
paths** for the engine to enumerate and solve.

---

## 4. Random tester

`random_test.py` is the standard "fuzz" comparison: it draws
`num_inputs` integers uniformly from the input range using Python's
stdlib `random.Random(seed)` (Mersenne Twister, fully reproducible) and
runs the host function on each one.  All (branch, taken) edges observed
are accumulated in the same `BranchTracker` as for the symbolic engine,
so the two methods are scored against an identical ground truth.

---

## 5. Experiment

Driver: `experiment.py`.  Configuration (held constant for both
methods, per the task brief):

* **Seeds**: `42`, `123`, `9999`  (≥ 3 different seeds).
* **Random inputs per seed**: `1 000`  (≥ 1 000).
* **Input range**: `[-1 000 000, 2 000 000]`.
* **Branches tracked**: 5  (10 edges total).
* **Magic branch**: `B0` with magic value `12345`.

## 6. Why random misses the magic branch

The probability of any single uniform draw in `[-10⁶, 2·10⁶]` hitting
`12345` is `1 / 3 000 001 ≈ 3.33 × 10⁻⁷`.  Consequently:

* P(hit in 1 000 draws) ≈ 1 − (1 − 1/3 000 001)¹⁰⁰⁰ ≈ **3.3 × 10⁻⁴**
* P(hit in 3 000 draws, the aggregate of 3 seeds) ≈ **1.0 × 10⁻³**

i.e. random testing has a 99.7 % chance of missing the magic branch in
a single seed and a 99.9 % chance of missing it across all three seeds.
This is exactly what we observed.  Note the KLEE paper itself reports
the same kind of behaviour in §5.2.4: "satisfying even a single 32-bit
equality requires correctly guessing one value out of four billion.
Correctly getting a sequence of such conditions is hopeless."

By contrast, the symbolic engine asks Z3 to satisfy the constraint
`x == 12345` directly, so it always finds the magic witness regardless
of how unlikely it is in absolute terms.  Path-constraint solving, not
guessing, is the lever.

---

## 7. Conclusion

1. **Symbolic execution achieves (effectively) complete branch
   coverage on this function** — 10 / 10 = 100 % on every seed, with
   only 13 concrete test cases (one per feasible path).  Crucially it
   reaches the hard-to-reach magic branch by **systematically solving
   the path constraint `x == 12345`** rather than by guessing.

2. **Random testing reliably misses the magic branch** in this budget:
   9 / 10 = 90 % coverage on every seed, with the B0-true edge
   unexercised.  Across 3 seeds × 1 000 draws = 3 000 inputs it still
   never sampled the magic value.  This is consistent with the
   theoretical hit probability of ≈ 3.3 × 10⁻⁴ per seed and demonstrates
   that **branch coverage is meaningfully lower (90 % vs 100 %)** and
   the rare branch is **unreachable** by random sampling in any
   practical test budget.

3. The result **reproduces the central empirical claim of the KLEE
   paper** — that constraint-based path enumeration can discover deep,
   low-probability behaviours (here, the "rare / magic" branch) that
   random or manual testing systematically overlook, while using
   orders of magnitude fewer test cases.
