# Z3 求解 N-皇后问题的规模扩展性

## 实验目的

考察 z3-solver 求解经典 N-皇后问题（N-Queens）时，求解时间随规模 `N` 的变化规律。
唯一自变量是 `N`；编码方式、求解器配置、随机种子、超时上限全部固定。

## 环境

- Python 3.11.2
- z3-solver 4.16.0.0（`pip show z3-solver` 验证）
- CPU only，未使用 GPU
- 单进程、串行运行

## 编码（固定不变）

按 task.md 要求，使用「每行一个 Int 列号」的纯整数编码：

- 变量：`Q[i]`，`i = 0 .. N-1`，类型 `z3.Int`，域 `0 ≤ Q[i] ≤ N-1`（用 0 起始列号；等价于 task.md 所说的 1..N 平移一个常数）。
- 列互不相同：`z3.Distinct(Q[0], Q[1], …, Q[N-1])`。
- 对角线互不相同：对所有 `i < j`，
  - `Q[i] - Q[j] != j - i` （主对角线）
  - `Q[i] - Q[j] != i - j` （副对角线）
- 这等价于 `|Q[i] - Q[j]| ≠ |i - j|`，是 N-皇后问题的标准 SMT 编码。

每条约束是 Int 域上的线性不等式，因此底层走 z3 的整数差分逻辑（simplex-based）求解器，再加上一个 DPLL(T) SAT 引擎处理 Distinct 等组合结构。

## 求解器设置（固定不变）

- `z3.set_param("smt.random_seed", 1234)` —— 固定随机种子。
- `z3.set_param("timeout", 120_000)` —— 120 秒硬上限（task.md 建议值），所有 `N` 均在上限内成功返回。
- 其余参数保持 z3 4.16 默认值（默认 tactic，未指定 `smt.phase`、`sat.phase`、`sat.restart` 等）。
- 对每个 `N` 都重新建一个 `z3.Solver()`，不重用任何状态。

## 运行协议

- 对 `N ∈ {8, 10, 12, 15, 20}`，每个规模都新建编码 → `solver.check()` → 取第一个解。
- 计时方法：`solver.check()` 之外用 `time.perf_counter()` 取墙钟时间差；计时起点在 `Solver` 构造完成之后、第一次 `check()` 之前；这样测得的是「建模后实际求解时间」，与「Python 启动 + z3 模块加载 + 编码构造」无关。
- 报告「第一次解」耗时 = `solver.check()` 墙钟时间。
- 完整性自检：对 z3 返回的解，独立的 Python 函数会验证列、两条对角线全部互不相同（见 `nqueens_experiment.py: verify_solution`）。

## 主要结果（固定随机种子 = 1234）

| N  | 结果 | 第一次解耗时 (s) | 自检合法 |
|----|------|-----------------:|----------|
|  8 | sat  |          0.0047 |     ✓    |
| 10 | sat  |          0.0072 |     ✓    |
| 12 | sat  |          0.0247 |     ✓    |
| 15 | sat  |          0.4468 |     ✓    |
| 20 | sat  |          0.3423 |     ✓    |

观察：

- 所有 5 个规模都在远低于 120 s 上限内完成（最大 0.45 s）。
- N=8 → N=15 单调增长：4.7 ms → 7.2 ms → 24.7 ms → 446.8 ms，**约 95×** 的跨度。
- **N=20 反而比 N=15 略快**（342 ms vs 447 ms）。这并不矛盾——它反映的是 SMT 求解器在「第一个解」语义下的搜索本质：z3 找到的是它在搜索树里**先撞到**的那一个解，依赖搜索启发式、随机种子、变量序号。固定种子是确定性的，但 N 大一点反而可能更快撞到一个早期可满足的分支。
- 验证过的第一个解：
  - N=8 : `[3,1,7,5,0,2,4,6]`
  - N=10: `[4,8,1,3,7,0,2,5,9,6]`
  - N=12: `[5,2,4,1,8,11,9,6,3,0,10,7]`
  - N=15: `[9,4,1,11,8,12,0,13,3,10,6,2,14,5,7]`
  - N=20: `[4,8,11,16,6,0,14,17,9,2,13,7,3,1,19,12,5,15,10,18]`

## 多随机种子的方差估计

为判断上面那条「N=20 比 N=15 更快」是不是噪声，我用 10 个不同种子（`{11, 23, 37, 53, 71, 97, 123, 257, 509, 1024}`）重复跑同一份编码。结果：

| N  | min (s) | median (s) | mean (s) |  max (s) |  max / min |
|----|--------:|-----------:|---------:|---------:|-----------:|
|  8 | 0.0038  |    0.0041  |   0.0041 |   0.0047 |       1.2× |
| 10 | 0.0049  |    0.0078  |   0.0070 |   0.0093 |       1.9× |
| 12 | 0.0129  |    0.0441  |   0.0472 |   0.1116 |       8.6× |
| 15 | 0.0375  |    0.1140  |   0.1014 |   0.1855 |       4.9× |
| 20 | 0.1468  |    0.2928  |   0.3785 |   0.8609 |       5.9× |

关键发现：

- **「第一个解」耗时随 N 增大而方差显著放大**。N=8 时 max/min ≈ 1.2×，基本是墙钟抖动；N≥12 时 max/min 跳到 5–9×，求解时间越来越依赖「先撞上哪一支」搜索树。
- 中位数随 N 单调增长：0.0041 → 0.0078 → 0.0441 → 0.1140 → 0.2928 s。
  - N=8 → N=20：约 **72×** 增长，远超多项式（2.5× 的 N 比）。
  - N=10 → N=20：约 **37×** 增长。粗略拟合 log(time) 与 N 接近线性，即求解时间随 N 呈**指数级**增长。
- 这与 N-皇后问题解的个数的增长一致：`Q(N) ≈ N! / e^(0.81·N)` 量级。即便只求一个解，在最坏情况下搜索量也大致按 `Q(N)` 的某个根/分支因子增长。

## 结论

1. 在「每行一个 Int + Distinct + 对角线不等式」这一固定编码下，z3-solver 4.16 对 `N ∈ {8, 10, 12, 15, 20}` 都能在远低于 120 s 超时上限内给出第一个合法解。
2. **求解时间随 N 呈近指数增长**：中位数耗时从 N=8 的 4 ms 涨到 N=20 的 293 ms，跨越约两个数量级。粗略外推，再大一两个 N 就开始需要秒级甚至十秒级。
3. **「第一个解」的耗时方差随 N 快速放大**（N=8 时 ~1.2×，N=20 时 ~6×），所以单次测量不可靠；固定随机种子只能稳定一条「剧情」，不能稳定耗时。要刻画真实扩展性，至少需要 5–10 次重复取中位数或最小值。
4. 对 N=20 这种规模，纯 SMT 编码就已经不算轻松——若要继续往上推（25、30…），要么换编码（如位向量 + 卡片约束、或纯 SAT 编码 `Q[i]·N + i` 一对一比特），要么加对称性破缺（比如 `Q[0] < N/2`）来削减搜索空间。
