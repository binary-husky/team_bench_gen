# 可行解 (sat) vs 最优解 (optimize) 对比

## 1. 实验设置

**固定问题实例**：5 jobs × 3 machines 的 job-shop 调度问题，每 job 3 道工序（共 15 道工序），全部为整数变量。

| Job | 工序顺序（machine, duration） |
|-----|-------------------------------|
| J1 | M1(2) → M2(3) → M3(1) |
| J2 | M2(1) → M3(2) → M1(3) |
| J3 | M3(3) → M1(2) → M2(2) |
| J4 | M1(1) → M3(2) → M2(3) |
| J5 | M2(2) → M1(1) → M3(2) |

**约束**（两种模式下完全相同）：
- `s_{j,k} >= 0`（非负开始时间）
- 工序顺序：`s_{j,k+1} >= s_{j,k} + d_{j,k}`
- 机器互斥（pairwise disjunction）：同机两道工序 `a, b`，要么 `a + d_a <= b`，要么 `b + d_b <= a`
- `makespan >= 0` 且 `makespan <= HORIZON = 30`

**决策变量**：每道工序的整数开始时间 `s_{j,k}`；以及上界变量 `makespan`。

**目标**：
- (a) SAT 模式：仅需返回任意可行解，记录其 `makespan` 值；
- (b) Optimize 模式：`minimize(makespan)`。

**自变量**：求解模式（sat vs optimize）。

**环境**：CPU only；`z3-solver` Python 绑定，版本 `4.16.0`。

**重复次数**：每种模式各跑 5 次，取均值/最小/中位耗时。

## 2. 实验结果

| 指标 | SAT 模式 (`Solver`) | Optimize 模式 (`Optimize`) |
|---|---|---|
| 状态 | `sat` × 5 | `sat` × 5 |
| 目标值（makespan） | **29** | **11**（最优） |
| 平均耗时 (s) | 0.00208 | 0.01513 |
| 最小耗时 (s) | 0.00184 | 0.01267 |
| 中位耗时 (s) | 0.00187 | 0.01616 |
| 耗时比（optimize / sat） | — | ≈ **7.3×** |

### 2.1 SAT 模式第一个解（makespan = 29）

`J1: s=(0,2,5)`；`J2: s=(5,6,8)`；`J3: s=(14,17,19)`；`J4: s=(11,12,21)`；`J5: s=(24,26,27)`。
所有 5 次运行给出**完全相同**的调度（每次 solver 从全新状态开始，但搜索路径恰好收敛到同一可行解）。

### 2.2 Optimize 模式最优解（makespan = 11）

以第一次运行的最优解为例：
`J1: s=(0,2,5)`、`J2: s=(5,6,8)`、`J3: s=(0,4,6)`、`J4: s=(2,3,8)`、`J5: s=(0,3,8)`。
5 次运行给出**不同的最优调度**，但 `makespan` 恒为 11（OPT 求解器在找到最优值后会寻找其他最优模型）。

### 2.3 最优性验证

每台机器的**总加工时间**是该实例 makespan 的下界：

| Machine | 总加工时间 |
|---|---|
| M1 | 9 |
| M2 | **11** ← 瓶颈 |
| M3 | 10 |

`max(9, 11, 10) = 11`，与 Z3 Optimize 给出的最优值 11 一致 → **11 确为全局最优**。

## 3. 对比与结论

1. **目标值差距悬殊**：SAT 模式仅追求可行，碰巧给出 makespan = 29 的解，离最优 11 差 **18 个时间单位**（最差解比最优解长约 **2.6 倍**）。这正是「feasible ≠ optimal」的典型体现。

2. **耗时差距温和但稳定**：Optimize 模式平均 ~15 ms，约为 SAT 模式 ~2 ms 的 **7 倍**。差距主要来自 Optimize 内部会反复收紧上界（依次证明 makespan ≤ 30, 29, …, 12, 11），本质上要做 O(opt) 次 SAT-like 检查。绝对耗时仍极小（毫秒级），说明该实例对 Z3 而言属于 trivial。

3. **稳定性的差异**：
   - SAT 模式 5 次运行几乎相同的解（同一可达解）→ 求解器初始决策对该问题空间收敛一致；
   - Optimize 模式 5 次运行给出 5 个**不同的**最优解，但目标值一致 → Z3 在多个最优模型间自由切换。

4. **方法论启示**：
   - 当问题存在**明显的瓶颈资源**（这里是 M2 的总加工时间 = 11），其和直接成为强下界，可与 Z3 Optimize 的结果互相验证；
   - 对实际调度/资源分配问题，仅 `check-sat` 返回的「可行解」往往**远远不是最优**，应优先用 `Optimize`（或内置的 `minimize`/`maximize` 软约束）；
   - 即使在本例这样的小规模实例上，optimize 的额外代价也只是毫秒级，**几乎总是值得**——除非模型构造本身昂贵或问题规模极大。
