# Chord 后继列表（successor list）在失效下保证路由正确性的机制

> 来源：Stoica et al., "Chord: A Scalable Peer-to-Peer Lookup Service for Internet Applications", SIGCOMM'01 — 重点参考 §4.4 (Node Joins), §5.2 (Failures and Replication), Theorem 7 与 Theorem 8 及其后关于 r = O(log N) 的论述。
>
> 本文仅基于论文内容做逻辑推演，不进行实验。

---

## 1. 后继列表让路由在直接后继失效时仍能正确转发的机制

### 1.1 正常路由对后继表的最低要求

Chord 把节点和键都映射到大小为 $2^m$ 的同一标识环上，按顺时针把每个键 k 分配给第一个节点 `successor(k)`。**仅靠每个节点 n 的 `successor` 指针（即直接后继）就足以让 `find_successor` 的迭代式转发维持正确**：每当 n 收到请求，它就把请求顺时针交给自己的直接后继，直到到达 `successor(k)`。这一条不变式就是 §4.3 中定理 2 的关键：转发只会单调朝目标靠近。

### 1.2 引入后继列表的冗余结构

§5.2 指出，节点一旦失效，持有失效节点作为直接后继的节点 n 必须能"自愈"——否则 n 既不能继续转发，也不能把自己的键迁出去。因此每个节点除了 `successor` 之外还维护一份"后继列表"，记下它在环上顺时针方向的 **r 个最近邻（nearest successors）**。

`stabilize()`（图 7）不断询问当前直接后继的 predecessor，据此刷新自己 successor-list 的内容；只要 n 还活着，它的列表就会按周期被修正到当前可见的真实后继序列上。

### 1.3 直接后继失效时仍能正确转发的关键性质

性质如下：**节点 n 的后继列表中第 1 项是 `successor(n)`，第 2 项是 `successor(successor(n))`，依此类推**——即列表就是 n 在环上顺时针 r 个最近活的（理想情况下）节点。

当 n 的直接后继（列表中第 1 项）失效时：
- n 通过 `stabilize()` 探测发现对方不再响应；
- n 把列表中第 2 项升级为自己的 `successor`，第 3 项提升为新列表的第 2 项，以此类推；
- 之后所有出站查询都用这份**已刷新的** successor-list 来向前转发。

这就是 §5.2 那句关键的论证：

> "If node n notices that its successor has failed, it replaces the first live entry in its successor list."

以及：

> "Ideally the lookups would be able to proceed, after a timeout, by another path despite the failure."

由此派生出**定理 7**：
> 若网络最初稳定，并令每个节点独立以概率 1/2 失效，若使用长度 r = O(log N) 的后继列表，则 `find_successor` 以高概率返回**最近存活的**后继。

直觉上：r = O(log N) 个候选中，每个仍活着的概率至少是 1/2，所以"全死"的概率被压到 $2^{-r} = 2^{-O(\log N)} = 1/\text{poly}(N)$——几乎不会发生；只要列表里还有至少一个活节点，n 就可以替换自己的直接后继，路由就仍然**沿着环上唯一正确的下一个活节点**前进。

### 1.4 因此"让路由继续正确"的关键性质是一句话

> **在环上从节点 n 顺时针方向出发，列表里至少存在一个仍然存活且 ID 严格大于 n 的节点——只要这一性质成立，n 就能从列表里挑出最近的那个活节点顶替成自己的直接后继，从而把请求朝 `successor(k)` 的正确方向继续转发。**

这本质上是"在环上做线性/右邻探测"在哈希环上的实例：r 越长，可容忍的连续失效段越长。

## 2. 路由真正断裂（节点"搁浅 / stranded"）的精确条件

把上面的性质取反，就得到"什么时候 n 真正被困住"。

### 2.1 节点 n 单方面的搁浅

n 在下列情况下会变成 stranded，无法继续转发任何查询：

- n 自己活着；
- 但 **n 的整个后继列表（r 个条目）所对应的节点全部同时失效**；
- 在 stabilization 还没来得及从更远的地方修复列表之前，n 找不到任何顺时针方向上的下一个活节点可作为自己的直接后继；
- 同时，n 的 finger table 中比后继列表更远的那部分条目本来就不能保证覆盖 n 与其真正直接后继之间的间隙（fingers 跳的是 2 的幂级距离，本就允许中间出现空段）。

这种情况的失败概率恰好是 $2^{-r}$，r 越大越不可能；r = O(log N) 时是 1/poly(N)。

### 2.2 全局意义上的"路由断裂"

不仅单节点会搁浅，全局路由还可能在以下任一条件下实质断裂：

1. **环被切成两半**：所有节点以某种方式被划分成两个互不相连的子环（典型是网络分区），每一边自己的 successor-list 都只指向自己这一侧，没有跨分区的指针；此时指向"另一半"的查询必然失败。
2. **节点被自身 ID 空间孤立**：某个节点 n 的顺时针方向上最近的活节点对 n 而言并不能真正"前进"（例如剩下的活节点与 n 之间的标识段根本不在 n 的视野里），n 就只能向自己绕回；
3. **fingers 也全失效**：即使 n 的后继列表里碰巧还有一个活节点，但如果更高层的 finger 表严重过时，路由会绕远甚至回退——论文 §6.4、§6.5 用 simulation 表明：在 1/2 随机失效下只要 successor-list 长度 r 足够，路由仍能工作，否则失败率会显著上升。

**精确地说，一个节点 n 何时变成 stranded：**
> **当且仅当在 n 的环上顺时针方向上，从 n 自身出发，连续 r 个最近的邻居节点全部失效（r 是 n 当前后继列表的长度），并且 n 自己也无法通过 finger table 找到任何 ID 严格大于 n 的活节点。**

注意，这个条件既要求"最近邻段清空"，也要求"更远处也无可用指针"——两者必须**同时**满足；只要其中之一还有活节点，n 都能找到下一跳。

### 2.3 边界：1/2 失效是"勉强能用"的临界点

论文 §5.2 的定理 7、8 的条件是**每个节点独立以概率 1/2 失效**，并且 r = O(log N)。这是**充分条件**，不是必要条件；如果失效概率 p 增大到远大于 1/2，那么 $(1-p)^r$ 随 r 衰减到不够快的程度，节点就会频繁被搁浅；如果 p 减小，则可以用更小的 r（开销更小）就能达到同样的失败概率。也就是说：

> **1/2 不是 Chord 的硬上限，而是论文用以演示"用 r = O(log N) 的 successor-list 即可容忍的典型强失效比例"。**

## 3. 为什么系统能容忍大约 ½（而不是更小或更大）节点失效

下面给出从论文出发的纯逻辑推导。

### 3.1 上界角度：1/2 来自于"独立二项失效 + 指数衰减"

考虑任意一个还活着的节点 n，它在 successor-list 中有 r 个候选邻居节点，每个候选仍存活的概率至少是 $1 - p$（$p$ 是单节点失效概率）。"整个列表全死"的概率为

$$
\Pr[\text{n stranded by list}] \;\le\; p^r .
$$

要使这一概率在所有 N 个节点上都压到 $\frac{1}{\text{poly}(N)}$，需要

$$
p^r \;\lesssim\; \frac{1}{\text{poly}(N)} \quad\Longrightarrow\quad r \;\gtrsim\; \frac{\log N}{-\log p}.
$$

- 若 p 很小（远小于 1/2），$-\log p$ 大，所以小 r 就够；
- 若 p = 1/2，$-\log p = 1$，需要 $r = \Theta(\log N)$，正好是论文 §5.2 取的值；
- 若 p > 1/2，$-\log p < 1$，需要 $r > \log N$，但 successor-list 的维护是 O(log N) 的（每 stabilization 一步做 O(log N) 次 RPC），r 再大就把 successor-list 维护本身变成新的瓶颈。

也就是说：**1/2 恰好是"维持 O(log N) 的 successor-list 长度，同时让搁浅概率仍可忽略"的最大失效比**——再大就要付出比 O(log N) 更大的代价。

### 3.2 下界角度：为什么不是某个更小的比例

直觉上，"半"看起来比"小比例"宽松得多，但有两个原因使得论文不去挑一个更小的数：

1. **定理 7 就是要展示一个强结论**：即便每个节点都有 1/2 概率失效——这是非常激进的、与正常运维毫无关系的失败模式——Chord 仍然能维持正确路由。这条定理的证明不依赖"大部分节点还活着"这种弱前提，因此其结果比"容忍 10% 失效"这种弱结论更有信息量。
2. **把 1/2 当作基准使 r = O(log N) 的开销可承受**：取 r = c·log N 时，每个节点多维护 c·log N 个邻居是 O(log N) 的空间和 stabilization 成本——这与 Chord 原本就承诺的每节点 O(log N) 存储与 O(log² N) 单次加入开销在同一量级，**没有引入新的渐近复杂度**。如果选一个更小的"容忍比例"作为基准，反而显得定理不够强；论文自然就用 1/2 这个对称的、优雅的数。

### 3.3 simulation 实验支撑

论文 §6.4、图 11 的 simulation 给出直观验证：在 $N = 2^{12}$ 的网络上让节点独立以概率 p 失效，等 stabilization 完成后查询失败率近似就是 p——也就是说 Chord 的"查找"机制对**失败节点没有放大**：失败节点不在线就找不到对应的键（这本就是定义上正确的失败）；只要键的真正负责者还活着，查询就能到达。换言之，"约 ½ 节点失效"在论文里就是**直接拿来做定理条件的那个比例**，并被实验确认不会引发额外的查找失败放大。

### 3.4 把三条合在一起：为什么是 ≈1/2

综合 §3.1–§3.3，可以给出一个清晰的总结：

- **机理上**：successor-list 提供 r 个候选后继，"全死概率"为 $p^r$；取 r = O(log N) 时该概率为 1/poly(N)。
- **复杂度上**：r = O(log N) 让 successor-list 维护成本与 Chord 原生开销同阶，没有引入新量级。
- **结论上**：p = 1/2 是 **"r = O(log N) 与 negligible 搁浅概率" 两条约束同时能成立的最大失效比**——再小（如 0.1）就太弱、定理没意义；再大（如 0.9）就超过 O(log N) 后继列表能压住的范围。
- **配合网络分区这一额外条件**（论文 §7 明确指出 Chord 不自行修复分区），"约 ½"指的是**均匀随机节点失效**下的容错，不是对抗性分区下的容错；后者任何 DHT 都无法仅靠 successor-list 解决。

## 4. 结论（直接回答题目三问）

1. **直接后继失效时为什么还能正确转发？**
   因为节点 n 维护了一张顺时针方向上 r 个最近邻的后继列表；n 的直接后继死了之后，n 通过 `stabilize()` 发现并把它列表里**下一个仍存活的邻居**升级为新的直接后继，所有后续的查询都基于这张已刷新的列表继续沿环顺时针前进。只要列表里至少还有一个 ID 比 n 大且活着的节点，转发方向就不会错（顺时针单调朝目标靠近）。

2. **路由何时真正断裂（节点 stranded 的精确条件）？**
   当且仅当 **n 的整个后继列表 r 个条目所对应的节点全部失效**，**并且** n 的 finger table 里也找不到任何 ID 严格大于 n 的活节点时，n 才被彻底孤立——它没有合法的"下一跳"可发请求，也无法把本地键迁出去，于是任何要跨过 n 这一段的查询都会失败。整体环若被网络分区切成两个互不相通的子环，则每一侧都可能局部出现这种 stranded 节点。

3. **为什么是 ≈1/2 而不是更小或更大？**
   因为定理 7 选择 r = O(log N)，那么"列表全死"的概率是 $p^r = 2^{-r} = 1/\text{poly}(N)$——只要 p = 1/2，这一表达式仍然随 r 指数衰减，足以让搁浅概率降到忽略不计；同时 r = O(log N) 没有超过 Chord 已承诺的 O(log N) 维护成本，因此 1/2 是 **"被 O(log N) 后继列表压住的、最大的均匀随机失效比"**。选更小的 p（如 0.1）则 r 可以更小，但定理就只是平凡弱结论；选更大的 p（如 0.9）则 $p^r$ 不再随 r = O(log N) 衰减到 1/poly(N)，必须 r > O(log N) 才能压住，那就突破了 successor-list 维护本身的渐近预算。论文正是把"≈1/2 失效、r = O(log N) 仍可工作"作为一条精确而强的定理呈现，因此这个比例是结果，不是任意挑的常数。
