# Chord lookup robustness to random node failures: r=1 vs r=16

## 实验设置

- **协议**: Stoica 等 (2001) "Chord: A Scalable Peer-to-peer Lookup Service for Internet Applications" — 关注 §5.2 的 successor-list 机制 (Theorems 7-8)。
- **进程内虚拟环**: 标识空间 2^16 = 65536。每个节点 id 通过 `random.sample(range(65536), N)` 从均匀分布中抽取（节点间平均间距 ≈ 65.5）。
- **固定参数**: N = 1000, 每组做 10 000 次随机键查找，自变量为 `r ∈ {1, 16}` 与 `f ∈ {0, 0.1, 0.2, 0.3, 0.5}`。每组用 5 个独立随机种子 (42–46) 重复实验，报告均值 / 标准差 / 最小值 / 最大值。
- **失败模型**:
  1. **先把 successor list 构造好**——在所有 N 个节点都还活着的时候，按 `r` 取顺时针最近的 `r` 个活后继（这模拟论文中的 "join-time" snapshot）。
  2. 然后再按比例 `f` 随机杀死节点，**不**运行任何 stabilization / fix-up。
  3. 这正是 §6.5 "Lookups During Stabilization" 描述的、修复尚未完成时的"中间状态"——successor list 是应对这种情况的唯一手段。
- **查找算法**（对应 Figure 4 + §5.2 的 successor-list 路由）：
  从随机活节点 `cur` 出发；取 `cur.successor_list` 中**第一个仍然存活**的节点 `next`；若 `K ∈ (cur, next]`（在 modular circle 上）则 `next` 即为 K 的负责节点；否则 `cur ← next`，迭代。环破裂时（successor list 中全部 `r` 个都已死）返回 `None`，算作失败。
- **度量**: 成功率 = 返回值等于当前 ring 上 K 的真正 live successor 的查询比例。每组 10 000 个 key 都先算 ground truth（当前活节点中第一个 ≥ K 的），再发起查找。

## 主结果（任务要求的 5 个 f 值 × 2 个 r 值）

| f (死节点比例) | 活节点数 | r = 1 成功率 | r = 16 成功率 |
|---:|---:|---:|---:|
| 0.0  | 1000 | **1.000** (σ=0.000, min=1.000, max=1.000) | **1.000** (σ=0.000, min=1.000, max=1.000) |
| 0.1  |  900 | **0.0073** (σ=0.0012)                  | **1.000** (σ=0.000)                  |
| 0.2  |  800 | **0.0039** (σ=0.0005)                  | **1.000** (σ=0.000)                  |
| 0.3  |  700 | **0.0022** (σ=0.0007)                  | **1.000** (σ=0.000)                  |
| 0.5  |  500 | **0.0011** (σ=0.0003)                  | **1.000** (σ=0.000)                  |

失败原因细分：`none_lookup` = 走到一半 successor list 全死，环破裂（占 r=1 失败的 100%，r=16 失败的 0%）；`wrong_lookup` = 返回了一个活节点但不是正确的（5 万次主实验里**一次都没有**——只要查到一个活的后继，返回的就是 ground truth，因为 successor list 里的活节点就是当前环上的真后继）。`avg_hops_on_success` ≈ 1.0（r=16 永远 1 跳成功，因为 16 项里几乎一定有活的）。

## 结论：successor list 长度把环的鲁棒性从"脆"变成"几乎不可摧毁"

1. **r=1 完全脆**。一旦 `f ≥ 0.1`，成功就从 100% 掉到 0.7%；继续增大 `f` 反而略降，因为可走的活节点数变少——这是因为 r=1 完全没有 fallback：每次 hop 都只看 `successor_list[0]`，它一死，链路立刻断。环里只要有任意一个活节点 `x`，使得 `x` 顺时针第一个真后继死掉，**所有经过 `x` 发起的查找都会卡死**。
2. **r=16 在全部任务要求的 f 下都是 100%**。直观上，查找在某节点 `x` 一步卡住的概率是 `f^r` = `0.1^16 ≈ 10^-16`（或 `0.5^16 ≈ 1.5×10^-5`）。即使环里一半节点都死，环上任一活节点的 successor list 16 项里几乎一定还有活的；取第一个活的就跳过去。每跳都是 O(1)。
3. **理论对照**：这正是 §5.2 Theorem 7 所预言的——长度 r 的 successor list 让 find_successor 在"每个节点以概率 1/2 独立失败"的模型下仍以**高概率**返回正确的最近活后继；Theorem 8 同时给出期望 O(log N) 跳数。我们实验里的 r=1 是失败容错的退化情形，r=16 = O(log N) 是论文推荐量级，二者的差距就是"是否启用 §5.2 机制"的差距。

## 扩展扫描（找 r=16 的失效阈值，不属于任务必需，仅为加深结论）

为了看清 r=16 在哪个 `f` 开始塌掉，额外跑了 `f ∈ {0.6, 0.7, 0.8, 0.9, 0.95}`：

| f    | r = 1    | r = 16   |
|-----:|---------:|---------:|
| 0.6  | 0.0007   | 1.000    |
| 0.7  | 0.0004   | 0.784 (σ=0.264, min=0.45, max=1.00)|
| 0.8  | 0.0002   | 0.275    |
| 0.9  | 0.0002   | 0.028    |
| 0.95 | 0.00006  | 0.0098   |

可见 r=16 一直撑到 ~60% 节点死掉仍然 100%；之后开始出现概率性的失效，原因是此时 successor list 的 16 项里偶尔会出现全部死掉的窗口（`0.6^16 ≈ 0.00028` 量级）；在 `f=0.7` 时这种现象开始可见，并随 `f` 增大而呈 `f^r` 增长。**这与 Theorem 7 中"以高概率成功"的"高概率"门槛一致**——r 越大，这个门槛越接近 1，可承受的 `f` 越大。

## 一句话结论

固定 N=1000、键查找 10 000 次、无 stabilization 的前提下，**successor list 从 r=1 增长到 r=16，把 Chord 查找对随机节点失效的鲁棒性从"f=0.1 即塌"提升到"f=0.5 仍 100%"**——这正是 §5.2 设计 successor list 的全部动机，也定量印证了 Chord 论文 Theorem 7/8 的预测。
