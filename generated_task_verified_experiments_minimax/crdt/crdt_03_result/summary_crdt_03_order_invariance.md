# CRDT Order-Invariance Experiment — Summary

**Goal.** Verify that state-based CRDTs' `merge` converges to a single final state
regardless of message delivery order or duplicate delivery, while a naive non-CRDT
control does not. CRDT correctness in asynchronous / out-of-order / at-least-once
networks rests on the merge operation being commutative, associative, and idempotent
(CvRDT conditions from Shapiro et al. 2011). The experiment demonstrates this
empirically.

**Source material.** Marc Shapiro, Nuno Preguiça, Carlos Baquero, Marek Zawirski,
*"A comprehensive study of Convergent and Commutative Replicated Data Types"* (INRIA
RR-7506, 2011) — in `crdt_material/crdt_shapiro_2011.pdf`. Specifications used:
Spec 6 (G-Counter), Spec 7 (PN-Counter), Spec 8 (LWW-Register), Spec 15 + state-based
U-Set reduction in §3.3.5 (OR-Set).

---

## 1. Implementation (in-process Python)

Implemented state-based CRDTs with a pure `merge` (no network, no threading):

| CRDT | Payload | `merge` semantics |
|------|---------|-------------------|
| **G-Counter** (Spec 6) | `vector[n]` of non-negative ints, one slot per replica | element-wise `max` |
| **PN-Counter** (Spec 7) | `(P[n], N[n])` | element-wise `max` on each; value = `sum(P) − sum(N)` |
| **LWW-Register** (Spec 8) | `(value, timestamp, replica_id)` | keep the entry with the largest `(timestamp, replica_id)` (lex order, replica id breaks ties) |
| **OR-Set** (§3.3.5) | `set[(element, unique_tag)]` of currently present pairs | set union |

`merge` returns a new state — it is pure and associative-friendly. These four are the
state-based counterparts that Shapiro et al. prove are CvRDTs.

**Naive non-CRDT controls:**
1. `NaiveLastWrite` — a plain integer counter where `merge(x, y) = y` (i.e. the
   incoming payload overwrites the local state). This is the textbook example of an
   eventually-consistent object that does **not** converge under out-of-order delivery.
2. `NaiveAccum` — `merge(x, y) = x + y`. Commutative and associative, but **not
   idempotent**: any duplicate message causes the value to be added twice.

---

## 2. Experimental setup

- **Fixed message set.** Deterministically generated with seed `2026` using 4 replicas,
  ~40 ops per replica; each op is one of the supported CRDT mutations plus a payload
  stream for the naive counters. Total messages per type (same for every scenario):

  | Stream | Count |
  |---|---|
  | G-Counter | 33 |
  | PN-Counter | 51 |
  | LWW-Register | 31 |
  | OR-Set adds | 25 |
  | OR-Set removes | 17 |
  | Naive last-write | 160 |
  | Naive accumulator | 160 |

- **Delivery scenarios.** For each scenario `s` (seeded by `2026 + 1`, scenario index):
  - take the message list,
  - produce a **random permutation** of its indices,
  - **duplicate ~30 %** of the messages (each duplicate chosen uniformly from the base
    list) so that each message is delivered at least once and ~1.3× on average.
- **Replay.** For each CRDT and each scenario, start from the empty initial state and
  fold the permuted+duplicated stream through `merge`; record the final state.

- **200 scenarios** were run (≥ 100 required).

All code is in `experiment.py`; algebraic-property micro-checks are in
`algebraic_check.py`; the figure is generated by `plot_results.py`. Raw per-scenario
final states are in `results.json`.

## 3. Results

### 3.1 Distinct final states (the headline metric)

| Type | Distinct final states (of 200) | Converges? |
|---|---:|---|
| **G-Counter** (CRDT)               | **1** | ✅ |
| **PN-Counter** (CRDT)              | **1** | ✅ |
| **LWW-Register** (CRDT)            | **1** | ✅ |
| **OR-Set** (CRDT)                  | **1** | ✅ |
| Naive last-write (control)         | 67    | ❌ |
| Naive accumulator (control)        | 35    | ❌ |

The four CRDTs all return a single final state regardless of the random
permutation + ~30 % duplication of the message stream. The naive controls
produce dozens of distinct outcomes — the last-write control's outcome depends
on which message happens to be processed last, and the accumulator's outcome
depends on how many duplicates the network happened to deliver.

![Distinct final states per CRDT / control](distinct_states.png)

### 3.2 Why this happens — algebraic-property check

`algebraic_check.py` drew 200 random triples of states per type and checked the three
CvRDT properties directly. Failures (out of 200 trials):

| Type | commutativity | associativity | idempotence |
|---|---:|---:|---:|
| G-Counter     | 0 | 0 | 0 |
| PN-Counter    | 0 | 0 | 0 |
| LWW-Register  | 0 | 0 | 0 |
| OR-Set        | 0 | 0 | 0 |
| NaiveLastWrite | **198** | 0 | 0 |
| NaiveAccum    | 0 | 0 | **200** |

The CRDTs are all three. `NaiveLastWrite` is non-commutative (because overwriting with
`a` then `b` ≠ overwriting with `b` then `a`), so the final state depends on delivery
order. `NaiveAccum` is non-idempotent (`a ⊕ a ≠ a`), so any duplicate message
double-counts and the final state depends on duplicate count.

### 3.3 Worked examples from `results.json`

- **G-Counter final value (across all 200 scenarios):** 232
- **PN-Counter final value (across all 200 scenarios):** 22
- **LWW-Register final value (across all 200 scenarios):** `v2_4` — the message with
  the highest `(timestamp, replica_id)` always wins.
- **OR-Set final value (across all 200 scenarios):**
  `{'e2_5', 'e1_6', 'e0_0', 'e0_7', 'e3_6', 'e3_0', 'e3_5'}` — same set each time.
- **NaiveLastWrite:** 67 different integers, e.g. `0, 1, 2, 3, …, 99` (any of the 160
  message values can end up as the last one delivered).
- **NaiveAccum:** 35 different sums, ranging from 287 (no duplicates seen) to ~520
  (many duplicates seen) — wider than the "exact" sum 363 because each duplicate adds
  its delta a second time.

## 4. Conclusions

1. **All four CRDTs converge under any delivery order and any amount of duplication.**
   Across 200 random permutations + ~30 % duplicates of the same message set,
   G-Counter, PN-Counter, LWW-Register, and OR-Set each produced exactly **one**
   distinct final state. This is the empirical signature of a well-formed state-based
   CRDT whose `merge` is **commutative, associative, and idempotent** (a join-semilattice
   whose join is `merge`, per Shapiro et al. §2.3 / §3). Such a structure guarantees
   that any sequence of merges containing the same set of messages yields the same
   least upper bound, independent of order and of any duplicates.

2. **Naive non-CRDT counters fail to converge.** The naive last-write counter yields
   67 distinct final states (≈ 1/3 of the message-pool size) — its `merge` is not
   commutative, so its outcome is the message processed last. The naive accumulator
   yields 35 distinct sums — it is commutative but not idempotent, so duplicate
   delivery inflates the value.

3. **Practical implication.** In an asynchronous network with arbitrary message
   reordering and at-least-once delivery semantics (TCP retransmits, gossip, mobile
   networks, etc.), a CRDT is the minimal data-type construction that lets two
   replicas end up in the same state without coordination. A plain non-CRDT counter
   is fundamentally unable to do this: with non-commutative `merge` it is
   order-dependent, and with non-idempotent `merge` it is duplicate-dependent. The
   200-scenario study makes this gap quantitative and reproducible.
