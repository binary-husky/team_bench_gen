# HyperLogLog — 小基数区 raw 估计偏差与 linear-counting 修正

复现 FlFuGaMe07（《HyperLogLog: the analysis of a near-optimal cardinality estimation algorithm》）中的算法，在**同一组寄存器**上比较两种估计量：

- **(A) raw**：`Ê_raw = α_m · m² · (Σ_j 2^{-M[j]})^{-1}`，**不做任何范围修正**。
- **(B) corrected**：当 `Ê_raw ≤ 2.5·m` 且存在零寄存器（`V>0`）时改用 linear counting `Ê* = m·ln(m/V)`，否则 `Ê* = Ê_raw`（论文 §4 的小范围修正）。

## 实现

- Python/numpy，CPU only。
- 64-bit 哈希：splitmix64 终结器（依赖无关、确定性、64-bit 混合良好）。寄存器索引取哈希**最高 p 位**，`ρ = (剩余 (64−p) 位中前导零个数) + 1`，寄存器存 `ρ` 的逐元素 max。
- `α_m` 用论文标准值（`m≥128` 时 `0.7213/(1+1.079/m)`）。
- 固定设置：`p = 10`，`m = 1024`，阈值 `2.5·m = 2560`；`n ∈ {100,500,1000,1500,2000,2560,3500,5000,8000}`；每 `n` 用 **8 个随机种子**独立重复；记录有符号相对误差 `(Ê−n)/n`。

脚本：`hll04.py`（实验）、`plot.py`（作图）。图：`hll04_bias.png`。

## 结论要点

1. **raw 在小基数区出现明显正偏（高估），且偏差随 n 减小而急剧增大。**
   `n=100` 时 raw 高估约 **+687%**，`n=500` 约 **+102%**，`n=1000` 约 **+34%**，`n=1500` 约 **+14%**，`n=2000` 约 **+6%**。这正是 HLL 调和均值估计的固有缺陷：当 `n` 与 `m` 同量级时，许多寄存器仍为 0，`Σ 2^{-M[j]}` 被大量 `2^{0}=1` 项（零寄存器）抬高，导致 `1/Σ` 被压低、`Ê_raw` 被系统性高估；`n` 越小，零寄存器越多，高估越严重。偏差方向一致为**正**，量级随 n 单调下降。

2. **linear-counting 修正把整个范围内的偏差压到接近 0。**
   在 `n ≲ 2.5·m` 区段 corrected 的均值相对误差全部落在 **±1.6%** 以内（多为 ±1% 量级），与 raw 的 +6%~+687% 形成对照。linear counting 直接用零寄存器比例 `V/m` 估计 `−ln(V/m)·m`，在小基数区是无偏的，因此消除了 raw 的正偏。各点标准差（~1–3.5%）与该 `m` 下 HLL 的典型相对误差量级 `~1.04/√m ≈ 3.2%` 一致，说明残差已主要是随机噪声而非系统偏差。

3. **`n ≳ 2.5·m` 时二者基本一致，修正分支不激活。**
   `n=3500/5000/8000`（均 > 2560）时 LC 分支 0/8 激活，corrected ≡ raw，数值完全相同（+1.36%/+0.66%/+0.50%），均在 ~1% 内、无系统偏。`n=2560`（恰为阈值）是个过渡点：8 个种子中只有 2 个满足 `Ê_raw ≤ 2.5·m 且 V>0` 而走 LC 分支，其余沿用 raw，因此两条曲线在此处开始汇合。这符合预期：阈值 `2.5·m` 是论文为“切换到 linear counting”设定的上界，越过它后零寄存器已稀少、raw 估计本身进入渐近无偏区，无需修正。

4. **机理小结**：raw 的正偏源自“零/小值寄存器对调和均值的扭曲”，是 `n=O(m)` 时的结构性偏差而非随机误差；linear counting 在 `V>0` 时以零寄存器计数替代调和均值，恰好补偿该扭曲；当 `n` 远大于 `m`、`V≈0` 时扭曲消失，两估计量自然合流。实验在 `p=10` 下清晰复现了这一全谱。
