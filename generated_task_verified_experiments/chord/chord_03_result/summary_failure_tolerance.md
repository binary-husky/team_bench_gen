# Chord 路由对节点失效的鲁棒性：后继列表长度的作用

## 1. 实验目标

在进程内（无真实网络）的虚拟 Chord 环上，考察路由对**随机节点失效**的鲁棒性。
固定 **N = 1000** 个节点；每个节点维护长度为 **r** 的后继列表（successor list）。
自变量为失效比例 **f ∈ {0, 0.1, 0.2, 0.3, 0.5}** 与后继列表长度 **r ∈ {1, 16}**。
对每个 (r, f) 组合，随机失效占比 f 的节点，再从**存活节点**发起约 **1×10⁴** 次随机键查找，
统计「查找成功率」——即查找结果恰好等于「负责该键的存活后继」的比例。

依据材料：Stoica et al., *Chord: A Scalable Peer-to-peer Lookup Service for Internet
Applications*, SIGCOMM 2001，尤其 §5.2 Failures and Replication 与定理 7、8。

## 2. 方法与设置

**Chord 环与路由（忠实于论文 §4）**
- 标识空间 m = 20 位（2²⁰ ≫ N，避免 id 碰撞），节点 id 在环上均匀随机选取并排序。
- 每个节点持有 m 项 finger table：`finger[k] = successor(id + 2^k)`，k = 0…m−1。
- 查找采用论文图 4 的迭代式 `find_successor` / `find_predecessor` /
  `closest_preceding_finger`（从高到低扫描 finger，落在 (n, key) 区间内最大的那项，距离减半）。

**失效处理（忠实于论文 §5.2）**
- finger table 与后继列表均在**失效前**的完整环上构建，失效后**不重建**
  （模拟 stabilize 完成前的瞬态）。路由时**跳过**指向已失效节点的 finger
  （论文：失效 finger 的替代「easily found in the finger table entries preceding that of the failed node」）。
- 每个节点的**有效后继 = 其后继列表里第一个存活条目**
  （论文：节点发现后继失效时，用后继列表中第一个存活条目替换）。
  - r = 1：后继列表只有直接后继；它一旦失效，节点**没有任何**后继兜底。
  - r = 16：可跳过最多 16 个连续失效的后继，找到第一个存活者。
- 「负责键 k 的存活节点」= 环上第一个 id ≥ k 的**存活**节点（最接近键的存活后继），
  作为查找成功与否的真值（ground truth）。查找能返回它即记成功；路由卡死或超时记失败。

**固定参数**
N = 1000；m = 20；每个 (r, f, seed) 做 10 000 次查找；种子 seeds = {0,1,2,3,4}（结果取均值±标准差）；
对同一 (seed, f) 使用**同一组查询键/起点**跑 r=1 与 r=16，构成配对比较；CPU-only，单机。

> 公平性：r=1 与 r=16 拥有**完全相同**的 finger table，唯一差别是后继列表长度 r。
> 因此本实验干净地隔离出「后继列表」这一因素对鲁棒性的决定性作用。

代码：`chord_failure_sim.py`；原始数据：`results.json`；图：`failure_tolerance_plot.png`。

## 3. 结果

**查找成功率 随 失效比例 f 的变化（均值 ± 标准差，5 个种子）**

| f | r = 1（单后继） | r = 16（后继列表） | r=1 上界 1−f |
|------:|:---------------:|:------------------:|:------------:|
| 0.0   | **1.0000** ± 0.000 | **1.0000** ± 0.000 | 1.00 |
| 0.1   | **0.7840** ± 0.022 | **1.0000** ± 0.000 | 0.90 |
| 0.2   | **0.5792** ± 0.009 | **1.0000** ± 0.000 | 0.80 |
| 0.3   | **0.4217** ± 0.010 | **0.9999** ± 0.000 | 0.70 |
| 0.5   | **0.1583** ± 0.014 | **1.0000** ± 0.000 | 0.50 |

（r = 16 在全部 5×5×10⁴ = 250 000 次查找中仅出现 1 次失败：f=0.3、seed 4，故该格为 0.9999；
其余均为 1.0000。）

曲线见 `failure_tolerance_plot.png`：r=16 几乎贴在 1.0 的水平线；r=1 随 f 单调快速下坠，
且**全程低于** 1−f 参考线。

## 4. 分析：为什么 r=1 崩溃、r=16 几乎不受影响

**(a) f = 0 时两者都 100%。** 无失效时 finger table 完整正确，标准 Chord 路由必然命中负责节点。
两条曲线在这里重合，是正确性基线。

**(b) r = 1 的崩溃来自「有效后继」失去冗余，且失效会级联。**
- r = 1 时，节点唯一的后继指针就是它的直接后继。查找要正确「落地」，必须存在一个存活节点 p，
  使得 p 的**直接后继恰好等于负责节点**。环上这样的 p 唯一——即负责节点的**直接前驱**。
  它存活的概率只有 **1 − f**，这给出 r=1 成功率的上界（表中第 4 列）。
- 实际成功率还**低于** 1−f，并且 f 越大、缺口越大（f=0.5 时 0.158 ≪ 0.50）。原因是：
  节点的直接后继失效后没有兜底，`find_predecessor` 在逼近目标时一旦沿途 finger / 后继
  连续失效就会**卡死**，路由连通性本身被破坏——**失效会沿环级联**。这使 r=1 的下降略呈超线性。

**(c) r = 16 的稳健来自后继列表把「连续失效」压成小概率事件。**
- 后继列表让节点跳过最多 16 个连续失效的后继去取第一个存活者。
  一个节点的 16 个后继**全部**失效的概率为 **f¹⁶**：即便 f = 0.5 也只有
  0.5¹⁶ ≈ 1.5×10⁻⁵。因此有效后继指针几乎永不死亡，路由始终能向前推进、最终命中负责节点。
- 这正是论文定理 7 的直觉（原文）：
  *「a node's r successors all fail with probability (1/2)^r, so with high probability a node
  will be aware of, so able to forward messages to, its closest living successor.」*
  r = 1 对应 (1/2)¹ 的暴露面，r = 16 对应 (1/2)¹⁶——量级之差直接体现在成功率上。

**(d) 与论文定理的对照。** 定理 7 指出：初始稳定网络中使用长度 **r ≥ 2·log₂N**（N=1000 时 ≈ 20）
的后继列表，即便每节点以 1/2 概率失效，`find_successor` 仍以高概率返回键的最近存活后继。
本实验 r = 16（略小于 20 但接近）在 f = 0.5 下成功率 ≥ 0.9999，与定理预测高度吻合；
r = 1 远低于该阈值，则在 f = 0.5 时跌到 ~0.16。两者形成鲜明对比。

## 5. 结论

- **后继列表长度 r 是 Chord 路由抗失效能力的决定性因素**，而非 finger table 的冗余
  （两组实验共享相同的 finger table，仅 r 不同，结果却天差地别）。
- **r = 1**（仅直接后继）几乎没有容错：成功率随 f 近线性、甚至超线性下跌，
  f = 0.5 时仅约 **15.8%**；失效会沿环级联，破坏路由连通性。
- **r = 16**（后继列表）把连续失效压成 f¹⁶ 的极小概率事件，成功率在全部 f 下保持
  **≥ 99.99%**，即便一半节点失效也几乎无损。
- 在 f = 0.5 时，r = 16 相对 r = 1 的成功率提升约 **6.3 倍**（+84 个百分点）。
- 该结果定量印证了 Stoica 等人关于后继列表（定理 7/8）的设计：用 O(log N) 规模的额外
  每节点状态，换取在大量并发失效下仍能正确、高效地完成查找。
