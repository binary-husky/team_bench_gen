# z3 可满足性求解 vs 优化求解：目标值与时间对比

## 任务

对比 z3-solver 的两种求解模式在同一固定实例上的表现：

- **(a) SAT**：用 `z3.Solver` 找一个可行解（`check()` 返回 `sat` 即可，不施加目标）。
- **(b) Optimize**：用 `z3.Optimize` 对目标 `makespan` 做 `minimize`，求最优值。

自变量 = 求解模式（sat vs optimize）；问题实例、变量域、约束、z3 设置全部固定不变。

环境：z3-solver 4.8.12（Python 绑定），CPU-only，Linux。脚本见同目录 `experiment.py`。

## 实例（固定）

小型作业车间调度（job-shop scheduling），最小化 makespan：

- 5 个作业 × 4 台机器；每个作业含 4 道工序，按固定顺序经过 4 台机器。
- 决策变量 `s[j][k]`（整数 `>= 0`）为作业 j 第 k 道工序的开始时间；`makespan` 为整数。
- 约束：
  - 同作业内工序次序：`s[j][k+1] >= s[j][k] + dur[j][k]`；
  - 同机器上任意两道工序不重叠：`Or(s1+d1 <= s2, s2+d2 <= s1)`；
  - `makespan >= s[j][k] + dur[j][k]` 对所有工序。

具体 route/duration 见 `experiment.py` 中的 `JOBS` 表，全程不变。两种模式共享同一份 `base_constraints()`，唯一差别是 sat 只调 `Solver.check()`，optimize 额外 `opt.minimize(makespan)`。

## 结果

每个模式重复 7 次（剔除首次冷启动后取中位数，单位秒；makespan 为整数）。

| 模式 | 目标值 (makespan) | 求解时间 (中位数, warm) | 时间样本 (s) |
|---|---|---|---|
| (a) SAT 可行解 | **73** | **0.0052** | 0.0055, 0.0066, 0.0047, 0.0056, 0.0049, 0.0048, 0.0158 |
| (b) Optimize 最优解 | **25** | **0.0160** | 0.0083, 0.0157, 0.0147, 0.0115, 0.0163, 0.0204, 0.0248 |

- 可行解目标值 / 最优目标值 = 73 / 25 = **2.92×**（可行解远劣于最优）。
- 优化 / 可行 时间比 = 0.0160 / 0.0052 = **3.06×**（优化慢约 3 倍）。

## 结论与解释

1. **目标值差异巨大（2.92×）。** `Solver` 只要 `check()` 返回 `sat` 即停——它返回的是 DPLL(T) 碰到的第一个满足约束的赋值，对 makespan 毫不关心。在本实例里这个"随手"赋值把若干工序的开始时间排得很靠后（makespan=73），与最优 25 相去甚远。这说明**可行性求解不能当作优化的替代**：得到的是"可行"而非"好"。

2. **优化求解代价更高（约 3×）。** `Optimize` 在 SAT 内核之上叠加了目标导向搜索（本版本通过反复收紧 makespan 上界/分支判定最优性），必须既证明"存在 makespan=25 的解"又证明"makespan<25 不可行"，因此耗时显著多于一次纯可行性判定。代价换来的是 2.92× 的目标改善——以 3× 的时间换取近 3× 的质量提升，在本实例上性价比相当。

3. **两者关系。** 优化求解内部反复调用 SAT 求解作为子过程（材料中 Z3 的 DPLL(T) 架构：SAT 内核 + 理论求解器；Optimize 在此基础上加 maxsat/带界目标循环）。所以 optimize ≈ 多次带界的 sat + 最优性证明，这与"它比单次 sat 慢、但能给出最优值"的观测一致。

4. **方法学要点。** 若只需"是否可行/能否排出来"，用 `Solver` 即可，快且足够；若关心调度质量（makespan 最小），必须用 `Optimize`。把 sat 的可行解误当作"近优解"在本例会产生 73 vs 25 的系统性偏差。
