# 查询缓存（counter-example cache）对符号执行 z3 调用数的影响

## 1. 实验设置

轻量 z3 符号 / concolic 框架（`query_cache_exp.py`，Python + `z3-solver`，未装 KLEE 本体，仅 CPU）。

- **可计数求解接口** `SolverInterface.check(constraints)`：每次"判断 PC 是否可满足"都经过它。
- **(A) 无缓存**：每次都调用 z3，`z3_calls == total_queries`。
- **(B) 带简单缓存**：以 **PC 约束集合（dedup 后的 `frozenset(str(constraint))`）** 为键缓存 `SAT/UNSAT` 结果与模型（反例）；命中即不调用 z3。`z3_calls == 不同查询数（即未命中数）`。
- **探索方式 = concolic（DSE）**：对每条执行路径上的每个分支，发出一个"翻转查询"`prefix_PC ∧ ¬taken_branch`。共享前缀的翻转查询在不同执行间反复出现 → 精确键命中；含重复子约束时，整段查询会塌缩成同一个键。
- **可比性保证**：固定 z3 的 `smt.random_seed`，使模型生成确定性；因此 (A) 与 (B) 走完全相同的查询序列与执行路径（断言 `total_queries` 与 `n_executions` 逐种子一致，已通过 `SANITY_OK`）。
- **≥3 种子**：种子 `1, 2, 3, 7`（共 4 个）控制初始随机输入与 worklist 弹出顺序。
- **被测 toy 函数**：`get_sign`、`classify_triangle`、`k_independent_ifs(k=4)`、`repeated_subconstraints(k=8)`（同一谓词 `x>0` 连续重复检查 8 次，模拟冗余守卫/展开循环/不变式复检）。

## 3. 结论

**带缓存的 z3 调用数显著少于无缓存，且与缓存命中率严格一致**：每个函数上 `(B) z3 调用数 = 总查询数 × (1 − 命中率)`，命中率从 ~67% 到 ~94%，对应 z3 调用下降 **约 3× 到 16×**（数倍～一个多数量级）。这与 KLEE 的核心信条一致——**"no query is the fastest query"**：不发给求解器的查询代价为零，缓存把大量本会重复打到 z3 的查询直接吸收掉。

**为什么符号执行的查询高度可缓存**（即命中率高的根本原因）：

1. **路径约束大量重复**。concolic/DSE 对每条路径在每个分支发"翻转查询" `prefix ∧ ¬branch`；任何走过同一前缀 `prefix` 并在同一分支翻转的执行，发出的是**完全相同的约束集合**。`k_independent_ifs` 中深度 j 处的翻转查询会被该前缀子树下的所有执行（共 2^(k−j) 条路径）重复发出 → 一次实算、多次命中（实测 136 查询 → 30 个不同查询，命中率 77.9%）。

2. **互为子集 / 超集，约束集合去重后塌缩**。PC 是约束的**集合**：同一谓词被反复加入会被去重。`repeated_subconstraints` 把 `x>0` 连查 8 次：全 T 路径在第 0 步翻转键为 `{¬(x>0)}`，第 1..7 步翻转键全部塌缩为 `{x>0, ¬(x>0)}`（同一个 UNSAT 集合）；全 F 路径的翻转键同样塌缩到 `{x>0}` 与 `{x>0,¬(x>0)}`。于是 48 个查询只有 3 个不同键，命中率 93.8%、z3 调用从 48 降到 3（16×）。这正是 KLEE 反例缓存利用的"子集/超集 + 去重"结构：超集已知 UNSAT 则子集免算；同一前缀的 SAT 模型可在更深查询中复用为反例。

3. **结构使然，非偶然**。符号执行的查询由"前缀约束 + 单个分支谓词"构成，前缀沿路径树高度共享、谓词集合有限，因此查询空间远小于执行次数；只要按约束集合缓存，重复率天然很高。真实程序里的冗余守卫、展开循环、跨状态复检、跨测试用例的同路径复走都会进一步放大这一效应——这也是 KLEE 把查询缓存列为关键工程优化的原因。

**局限**：本实验的缓存是**精确键 + 去重**的简单版本（按 task 指定）；KLEE 真实缓存还显式做子集(UNSAT)/超集(SAT 模型复用)匹配与反向蕴含，能在独立分支树（各前缀互不相同）上进一步榨取命中。即便如此，仅"精确键 + 去重"已在 4 个 toy 函数上把 z3 调用降到 1/3 ~ 1/16，验证了查询缓存的核心价值。
