# 为什么 Cuckoo Filter 必须为每个键保留**至少两个**候选桶，且**恰好两个**、由 `i2 = i1 ⊕ hash(f)` 联系就足够

> 任务要求：仅凭逻辑推理回答（不做实验）。结论与完整推导如下。依据为给定材料 *Fan et al., "Cuckoo Filter: Practically Better Than Bloom", CoNEXT'14*（下称「论文」）。

---

## 0. 结论先行（TL;DR）

1. **必须 ≥2 个候选桶**：cuckoo hashing 的灵魂是「踢出 / 搬迁（kick-out / relocation）」——当新键的两个桶都满时，把已有键搬到它的**另一个**家。**搬迁的存在性**依赖于「每个键有一个备选位置」。只有一个桶时，键无处可搬、被踢出的键也无家可归，搬迁机制从定义上就不存在；插入会在「该键的唯一桶已满」这一刻**立即、不可恢复地失败**，整个 filter 失去的核心优势是 **高表占用率 α（~95%）**，进而失去 cuckoo filter 相对 Bloom filter 的**空间效率**优势（`C = f/α`，α 崩塌 ⇒ bits/item 翻倍）。
2. **恰好两个就够**：标准 cuckoo hashing 本身就是「每键两桶」，两桶已足以让占用率在 b=4 时达到 ~95%（论文 §5.1 的实测）；多于两个只会增加每次查询的探测条目数 → 拉长指纹 → 增加空间（论文式 5/6），得不偿失。
3. **`i2 = i1 ⊕ hash(f)` 为何足够（且是关键）**：因为只存指纹不存原始键，被踢出的键无法用 `hash(原键)` 重新算出它的另一桶。XOR 是**自反**算子（`a ⊕ b ⊕ b = a`），于是「当前桶索引 i + 指纹 f」即可唯一反推出另一桶 `j = i ⊕ hash(f)`，**无论 i 是 i1 还是 i2**。这一双向可恢复性，让 filter 在**从不持有原始键**的情况下完成任意长度的搬迁链。

## 1. 为什么【必须】至少两个候选桶（一个为什么不行）

### 1.1 结构本质：cuckoo 的「搬迁」要求每个键拥有「另一个家」

先回到 cuckoo hashing 的定义（论文 §2.2）：每个键 x 由两个哈希函数映射到**两个候选桶** `h1(x)`、`h2(x)`。插入 x 时，若两桶之一有空位则直接放入；**若两桶都满**，则挑一个桶、把其中的旧键 a 踢出，并**把 a 重新插入到 a 自己的另一个候选位置**；这会连锁地踢出下一个键，直到找到空位或达到踢出上限（论文实现里 500 次）。

搬迁链之所以能展开，根本前提是：**被踢出的旧键必须还有一个备选桶可去**。这是一条「 augmenting path（增广路径）」：在「cuckoo 图」里，桶是顶点、每个键是连接它**两个**候选桶的一条**边**；搬迁 = 沿着这条边把键移到对端顶点，再从对端沿另一条边继续，如此交替延伸，直到撞上一个空顶点。

> **结构判据**：cuckoo 的搬迁在图论上等价于「在以键为边、桶为顶点的二部图上寻找增广路径」。要让键能从一个桶移动到**另一个**桶，每个键**至少**要连接两个不同的桶顶点——即每个键至少两个候选桶。这是增广路径能「动起来」的**拓扑最小要求**。

### 1.2 只有一个候选桶时：插入何时、以何种方式失败

若每个键只剩一个桶（`i = hash(x)`，无 `i2`），则：

- **搬迁链整体坍缩**：论文 Algorithm 1 里唯一能创造空位的代码段就是搬迁循环 `for n = 0; n < MaxNumKicks; n++ { … i = i ⊕ hash(f) … }`。单桶时，被踢出的键**没有第二个桶**，`i ⊕ hash(f)` 无意义（指回同一个已满的桶），循环第一步就是死路。整个 kick-out 机制不复存在。
- **失败时机与方式**：插入 x → 算出唯一桶 `i` → 若 `i` 已满（所有 b 个 entry 占满），则**立刻返回 Failure**，没有任何「换个桶试试」或「踢别人」的分支可走。失败是**即时、确定、不可恢复**的（除非扩表）。
- **失败发生得多早**：没有搬迁 = 退化为「静态哈希」，占用率被「无重排下的装箱上限」死死卡住。论文 §5.1 给出 `k=2, b=1` 时 α≈50%；而 `k=1`（单桶）还要更糟——本质上退化成「一旦某个桶被要求装第 b+1 个键就停」，这是生日悖论型的早期饱和，α 随表增大趋于很低。论文中 dl-CBF 用 **4 个桶**且**不搬迁**也只能到 ~80%（§7.2/§7.3），单桶只会更差。

### 1.3 失去的核心优势：高占用率 ⇒ 空间效率

论文式 (4) 给出每键均摊空间 `C = f / α`。整个 cuckoo filter 之所以能「比 Bloom filter 更省空间」（论文卖点之一，§1/§6，尤其 ε<3% 时），**完全建立在 α≈95% 这个高占用率上**。

单桶设计让 α 从 ~95% 跌到很低，于是：
- `C = f/α` 直接放大数倍，每键成本暴涨；
- cuckoo filter 对 Bloom filter 的**空间优势（论文四大优势之 4）被彻底抹平甚至反转**；
- 连带失去「高负载下仍能动态插入」的能力——插入在很低负载就开始大量失败，表必须频繁扩容（论文 §6「Maximum Capacity」问题被放大成常态）。

至于另外三大优势（动态增删、查询性能、实现简单）：单桶下「删/查」仍可做（删=删指纹，查=查一个桶），但**支撑「省空间」的高占用率这一地基没了**。所以「核心优势」精确地说就是：**由搬迁带来的高表占用率，及其衍生出的空间效率**——而这正是 cuckoo（杜鹃）之所以叫 cuckoo 的全部理由。

> 小结：**至少两个候选桶，是「搬迁」这一机制能被定义、能被执行的拓扑最小条件。** 一个桶 ⇒ 无备选 ⇒ 无搬迁 ⇒ 无高占用 ⇒ 退化成低占用静态指纹表，丧失 cuckoo filter 的立身之本。

## 2. 为什么【恰好两个】、且由 `i2 = i1 ⊕ hash(f)` 联系就足够

这一问有两个子问题要同时回答：(A) 两个为何够（够达到目标占用、够完成搬迁）；(B) 在「**从不存原始键**」的前提下，这两个桶之间的关系为何必须/为何能用 XOR 这么联系。

### 2.1 难点：只存指纹，被踢出的键「忘了自己是谁」

标准 cuckoo hashing（存完整键）搬迁时，能对被踢出的键 y 重新算 `h1(y), h2(y)`，因为 y 的原始键在手。但 cuckoo filter **只存指纹 f**（论文 §3：「the original key and value bits of each item are no longer retrievable」）。被踢出的 entry 里**只有 f，没有原始键**，所以**无法**用 `hash(原键)` 重算它的另一桶。这是论文 §2.2 / §3 反复强调的「standard cuckoo hashing techniques do not carry over」的核心障碍。

→ 因此，两个候选桶之间的**关系必须设计成「仅凭指纹 + 当前桶索引」即可互相推导**。否则搬迁就无法启动。

### 2.2 XOR 的自反性 ⇒ 双向可恢复 ⇒ 无需原始键即可搬迁

论文式 (1)：`i2 = i1 ⊕ hash(f)`，其中 `i1 = hash(x)`。

关键性质来自 **XOR 是自反（对合）算子**：`a ⊕ b ⊕ b = a`。据此：

- 由 i1 与 f 得 i2：`i2 = i1 ⊕ hash(f)`；
- 由 i2 与 f 反推 i1：`i1 = i2 ⊕ hash(f)`（因为 `i2 ⊕ hash(f) = i1 ⊕ hash(f) ⊕ hash(f) = i1`）。

也就是说，**「当前桶 i」与「指纹 f」用同一条公式 `j = i ⊕ hash(f)`（论文式 2）就能唯一得到「另一桶」，无论 i 究竟是 i1 还是 i2**。论文 §3 原话：XOR「ensures an important property: h1(x) can also be calculated from h2(x) and the fingerprint using the same formula」。

代入 Algorithm 1 的搬迁循环：取出被踢 entry 的指纹 f'，算 `i' = i ⊕ hash(f')`，把 f' 放到 i'——i' 必然是 f' 的**另一个**候选桶（由上面的对称性保证）；下一轮再对新被踢的指纹重复。**整条搬迁链从头到尾只用到「表里已有的指纹 + 当前桶下标」，从不触碰任何原始键。** 这就是「为何两个 XOR 相连的桶就足以在不存/不重算原键的前提下完成 kick-out」的精确答案。

> 注意区分两件事，避免混淆：
> - **搬迁可行性**只要求两个桶之间「可由指纹+当前索引互推」——XOR 的自反性满足这点（换成非自反的、例如模非 2 的幂下的加法，就不能用同一运算双向恢复，踢出的指纹会「找不到回家的路」）。
> - 论文 §3 里「**先 hash 指纹再 XOR**」是为了**均匀分布、避免聚簇**（8-bit 指纹若直接 `i ⊕ f`，踢出的键只会落在距 i 256 桶以内的局部，论文专门举例说明）。这是**占用率**层面的优化，**不是**搬迁可行性所必需的。两者不可混为一谈。

### 2.3 为什么是「恰好两个」而非更多

- **下界（不能少于 2）**：第 1 节已证——少于 2 则搬迁从定义上不存在。
- **上界（2 已足够，无需 3+）**：
  - **占用率已达标**：标准「双桶 cuckoo」即 `k=2`，在 b=4 时 α≈95%、b=8 时 α≈98%（论文 §5.1、§4、Figure 2）。两个桶已经把「power of two choices」的好处基本吃满；继续加桶对 α 的边际收益极小。
  - **加桶反而有害**：候选桶越多 ⇒ 每次查询要探测的 entry 越多 ⇒ 误判率上升 ⇒ 为维持同一 ε 必须用**更长的指纹** ⇒ 每键空间变大。论文式 (5)/(6)：`f ≥ log2(2b/ε)`，b 越大 f 越大。所以「多桶」是用空间换那一点点已经够用的占用率，净亏损。论文最终把默认配置定为 **(2,4)-cuckoo filter**（每键 2 桶、每桶 4 指纹，§5.1），正是「恰好两个」的最优性结论。
  - **图论最小性**：搬迁只需一条「键—桶」的边让键能在两顶点间移动；两个桶已是「让一个键能在桶顶点间迁移」的**最小可用图结构**。三个及以上桶会让每条键变成超边，既无必要，也破坏了 §2.2 那种简洁的「单条 XOR 关系双向恢复」性质（多桶时要额外记录「当前在哪个桶」才能枚举其余桶，丧失 minimality）。

> 小结：**两个，是「搬迁可行」所需的最少数；而两桶已足以把占用率推到 ~95%，再多只会拉长指纹、变费空间。** 配合 `i2 = i1 ⊕ hash(f)` 的自反性，搬迁链可在纯指纹世界里无限延伸。

## 3. 三者如何扣合成一个自洽的设计

把两个问题合起来看，cuckoo filter 的设计是一次「最小可行」的精巧闭环：

| 需求 | 由谁满足 |
|---|---|
| 高占用（≈95%）以省空间 | cuckoo 的搬迁机制 ⇒ **拓扑上要求 ≥2 桶** |
| 搬迁能启动（被踢键有备选家） | **恰好 2 桶**即满足（最小可用图结构） |
| 不存原始键也能算出备选家 | `i2 = i1 ⊕ hash(f)` 的 **XOR 自反性** ⇒ 双向可恢复 |
| 搬迁链任意长都能持续 | 同一条公式 `j = i ⊕ hash(f)` 反复适用，全程零原始键 |
| 不因多桶而变费空间 | 固定 2 桶 ⇒ 探测 entry 数固定 ⇒ 指纹长度最小 |

抽掉任何一环都会崩：
- 抽掉「第二个桶」→ 搬迁消失 → α 崩 → 空间优势消失（§1）。
- 抽掉「XOR 自反关系」（或换成非自反映射）→ 被踢指纹算不回另一桶 → 即便有两桶也无法搬迁 → 退化为「两个独立静态桶」，仍无高占用。
- 把「2」加成「3+」→ 占用率边际收益微乎其微，却因探测面变大而被迫拉长指纹 → 空间变差。

**因此：至少两个，是搬迁机制存在的拓扑下界；恰好两个、且由 `i2 = i1 ⊕ hash(f)` 这条自反关系联结，是「在永不存储或重算原始键的前提下完成指纹搬迁」的最小充分设计。**
