# CRDT 消息代价对比：state-based（G-Counter）vs op-based（CmRDT Counter）

> 任务 crdt_04：在 Python 进程内实现同一 increment-only 计数器的两种 CRDT 形态，
> 测量并对比它们的「每条消息字节数」与「总传输字节数」随操作数 M 的变化。
> 材料依据：Shapiro et al., *A comprehensive study of Convergent and Commutative
> Replicated Data Types*, INRIA RR-7506, 2011（`crdt_material/crdt_shapiro_2011.pdf`）。

## 1. 设计与依据（来自材料）

两种形态直接取自材料的规范：

- **op-based 计数器（CmRDT，Specification 5）**：payload 为单个整数 `i`，`increment`
  的 downstream 阶段执行 `i := i + 1`。每次本地 `increment` 只需广播一条**定长小消息**
  `{"src": i, "op": "inc"}`（即「副本 i 加 1」），经可靠因果广播送达后各副本累加。
  因 +1 满足交换律，其为 CmRDT。

- **state-based 增量计数器 G-Counter（CvRDT，Specification 6，vector 版）**：payload 为
  向量 `integer[N] P`（每个副本一项）；`increment` 执行 `P[g] := P[g] + 1`；
  `merge(X,Y)` 为逐分量 `max`。每次本地 `increment` 后，副本把**完整状态向量**
  （长度 N 的每副本计数向量）作为 payload 广播给其余副本，接收方逐分量取 max。

字节计量 = JSON 编码的实际字节数（`len(json.dumps(...).encode())`）：
- state 消息 = 裸向量 `json.dumps(V)`，大小随 N（条目数）**和** 计数值（数字位数）增长。
- op 消息 = `json.dumps({"src": i, "op": "inc"})`，i∈{0..4} 为单数字，**大小恒定 = 23 B**。

## 2. 实验设置（固定，见 task.md）

- 副本数 **N = 5**；M 均匀分摊到各副本（每副本恰好 M/N 次本地 increment）。
- **M ∈ {1000, 5000, 10000, 50000}**。
- 同步进程内模型、可靠即时投递：每次本地 increment 后立即广播一次。
  （此前所有广播均已即时送达，故广播者的本地向量 == 全局向量 V；
  这是最自然的进程内读法，结果因此近似确定，种子只影响操作的交错顺序。）
- **3 个种子**（seed ∈ {1, 7, 42}）随机化操作交错顺序，结果取均值（各种子间差异极小）。
- 「每条消息字节数」= 单条广播 payload 字节数；state 取 M 次广播的均值。
- 「总传输字节数」(主列) = 对 M 次广播各计一次 payload 字节之和。
  每次广播扇出到 N−1 = **4** 个对端，故**实际链路字节 = 主列 × 4**；
  两种形态同乘 4，比值与结论不变（下文另注）。

## 3. 结果表（每个 M 下两种形态的每条消息字节 + 总字节；3 种子均值）

| M（总操作数） | op 每条消息 (B) | op 总字节 (B) | state 每条消息 平均 (B) | state 每条消息 [min..max] (B) | state 总字节 (B) | state/op 总字节比 |
|---:|---:|---:|---:|:---:|---:|---:|
| 1 000   | **23**（常数） | 23 000    | 22.28 | [15 .. 25] | 22 279    | 0.969 |
| 5 000   | **23**（常数） | 115 000   | 24.46 | [15 .. 30] | 122 290   | 1.063 |
| 10 000  | **23**（常数） | 230 000   | 27.23 | [15 .. 30] | 272 273   | 1.184 |
| 50 000  | **23**（常数） | 1 150 000 | 29.45 | [15 .. 35] | 1 472 274 | 1.280 |

> 链路实际字节（×4 扇出）：例如 M=50 000 时 op = 4.60 MB、state = 5.89 MB；比值仍为 1.280。
>
> 示例 state payload：`[0,0,0,0,1]`→15 B；`[200,200,200,200,200]`→25 B；
> `[10000,10000,10000,10000,10000]`→35 B（位数随计数增长）。

## 4. 结论要点

1. **op-based 每条消息 O(1)，小而频繁。** op 消息恒为 23 B（`{"src": i, "op": "inc"}`），
   与 N、与计数大小均无关；总字节严格线性 `O(M·const)`：实测 23 000 → 1 150 000 B，
   恰为 23×M，M 从 1k→50k（50×）时总字节也精确 50×。

2. **state-based 每条消息 O(N) 且随计数增长，大而（相对）不频繁。** 每条消息携带长度 N=5
   的完整向量，单条字节随每个分量计数增大而增大（实测平均 22.28 B@1k → 29.45 B@50k，
   单条上限 15→35 B）。其单条消息显著大于 op 的常数 23 B（M≥5k 后即超过）。

3. **随 M 增大，state-based 总字节增长更快（每条都含不断增长的状态）。**
   op 总字节 = O(M·const) 严格线性；state 总字节 ≈ M × O(N·log(M/N))，每条消息系数随
   计数对数增长。实测 state/op 总字节比 **单调上升 0.969 → 1.063 → 1.184 → 1.280**；
   M 从 1k→50k（50×）时 state 总字节涨约 **66×**（22 279 → 1 472 274），明显快于 op 的 50×。
   小 M（≈1k）时因计数尚小、向量短，state 可略省（比值<1）；一旦计数上去，state 在
   「每条」和「总量」上都反超并持续拉开。

4. **权衡（材料 §2.4 的取舍在数据上的体现）：**
   - **高频小更新场景 → op-based 更省带宽**：每条 O(1) 常数消息，总量严格线性、
     不会随状态膨胀；适合点击计数、在线人数等增量密集型负载。代价是需要可靠因果广播，
     且要跟踪成员关系/去重（Spec 2/Spec 5 的 delivery-order 与去重集合 D）。
   - **低频大状态、或需断点续传 / 弱通道场景 → state-based 更简单、更稳健**：
     merge 幂等可交换，消息可丢、可乱序、可重复（材料 Proposition 2.1，仅需「最终可靠」
     通道），天然支持断点续传与后台反熵同步；不必维护因果元数据。代价是每条消息要发
     整个 O(N)（PN-Counter 更是 2N）向量、且随计数值增大，高频下带宽代价更高。

   一句话：**消息数随 M 线性增长时，op 把代价压在「条数」上（每条 O(1)），
   state 把代价压在「单条体积」上（O(N) 且随计数膨胀）；前者在高频小更新下省带宽，
   后者在低频/弱通道/断点续传下更简单可靠。**

## 附：方法与复现

- 代码：`crdt_04_experiment.py`；原始数值：`crdt_04_results.json`。运行 `python3 crdt_04_experiment.py`。
- 模型为同步进程内、可靠即时投递；state 广播 payload 取广播时刻的全局向量 V（G-Counter 完整状态）。
- 字节计量为 JSON 实际字节数；op 为定长（`{"src": i, "op": "inc"}` = 23 B），state 为裸向量 `json.dumps(V)`。
- 3 种子结果差异极小（<0.5%），印证「结果近似确定」。
